# PR 037 — Canvas Boundary Rules, Grid Snapping, and Keyboard UX Improvements

## Purpose

Make the floor map editor feel controlled and professional before adding bookable desks and booking flows.

## Why this PR exists

PR 035 introduced resize/rotate. After those changes, objects could be dragged completely outside the canvas, positions were unconstrained free-float numbers, and the grid was fixed at 50 px with no user control. This PR addresses those gaps.

---

## What changed

### 1. Canvas constants moved to `coordinateHelpers.ts`

`CANVAS_WIDTH` (1000) and `CANVAS_HEIGHT` (640) are now exported from `coordinateHelpers.ts` and imported everywhere they are needed, including `FloorMapCanvas.tsx` and `FloorLayoutPage.tsx`. This prevents the constants from diverging between the canvas renderer and the boundary-clamp logic.

### 2. Boundary rules (clamping)

New helpers in `coordinateHelpers.ts`:

| Helper | Purpose |
|---|---|
| `clamp(value, min, max)` | Numeric clamp; returns `min` for NaN |
| `clampObjectPosition(x, y, w, h, cw, ch)` | Clamp top-left so object stays inside canvas |
| `clampObjectTransform(x, y, w, h, cw, ch)` | Clip size to canvas, then clamp position |

**Policy:**
- Top-left `x` is clamped to `[0, canvasWidth − width]`.
- Top-left `y` is clamped to `[0, canvasHeight − height]`.
- If the object is larger than the canvas on either axis, it is anchored at 0 so it remains partially visible.
- Clamping uses the **unrotated** bounding box. A rotated object can have corners that exceed the canvas; precise rotated-boundary collision is deferred to a later PR.

Clamping is applied:
- After every drag (`handleObjectDragEnd`)
- After every transform/resize (`handleObjectTransform`)
- After every keyboard move (`handleCanvasKeyDown`)

### 3. Grid snapping

New helpers in `coordinateHelpers.ts`:

| Helper | Purpose |
|---|---|
| `snapToGrid(value, gridSize)` | Snap one value to nearest grid multiple; returns 0 for non-finite value |
| `snapObjectToGrid(x, y, gridSize)` | Snap both position axes |
| `snapSizeToGrid(w, h, gridSize, minSize)` | Snap size, never below `MIN_OBJECT_SIZE` |

**Rounding behaviour:** `snapToGrid` uses `Math.round`, which rounds half-values **up** in JavaScript. For example, with `gridSize=20`: a value of `10` snaps to `20` (not `0`), because `Math.round(10/20) = Math.round(0.5) = 1`. Document this to users if midpoint placement matters.

**Order of operations:** snap first → clamp second. This guarantees the final stored position is always inside the canvas after both transforms are applied.

**Drag:** both axes snap after drag-end.

**Keyboard movement:** only the axis that moved snaps. Pressing ArrowRight with `y=150` and `gridSize=20` leaves `y=150` unchanged. If both axes were snapped, `y` would jump from 150 to 160, which the user did not request.

**Transform (resize):** size snaps via `snapSizeToGrid`, then position snaps, then both are clamped via `clampObjectTransform`.

**Default state:**
- Grid visible: `true`
- Snap enabled: `false` (preserves free-movement behaviour for existing users)
- Grid size: `20 px`

### 4. Grid rendering update (`FloorMapCanvas.tsx`)

- `buildGridLines` replaced by a `useMemo` that depends on `showGrid` and `gridSize`.
- When `showGrid=false`, the memo returns `[]` — no grid lines are rendered.
- When `gridSize` changes the memo rebuilds with the new spacing.
- Available sizes: 10, 20, 40 px.

### 5. Canvas toolbar (`LayoutCanvasToolbar.tsx`)

A new component rendered above the lazy canvas:

| Control | Who sees it | What it does |
|---|---|---|
| Show Grid (Switch) | Everyone | Toggles grid visibility |
| Snap to Grid (Switch) | Editors only | Enables/disables snap |
| Grid size (ToggleButtonGroup) | Editors only | Chooses 10 / 20 / 40 px |

State is kept in `FloorLayoutPage` local state and is not persisted to the backend in this PR.

### 6. Keyboard UX (Option A — e.repeat guard)

`e.repeat` events (generated by held keys) are ignored with an early-return guard. This prevents PATCH flooding if the user holds an arrow key down.

- When **snap is disabled:** step = 1 px (Shift = 10 px).
- When **snap is enabled:** step = `gridSize`. Shift has no additional effect when snap is on.

The keyboard hint text updates based on snap state:
- Snap off: *"Tap arrow keys to move. Hold Shift for 10 px steps."*
- Snap on: *"Tap arrow keys to move by one grid step."*

The `e.repeat` suppression is intentional and documented. A debounced held-key option can be added in a future PR if needed.

### 7. Helper NaN / non-finite safety

- `clamp(NaN, min, max)` returns `min`. `Infinity`/`-Infinity` pass through `Math.min`/`Math.max` correctly (clamped to max/min respectively).
- `snapToGrid(NaN/Infinity/-Infinity, gridSize)` returns `0`. Non-finite or non-positive `gridSize` leaves the value unchanged.

These guards are defensive; in practice all inputs come from `parseFloat` on backend decimal strings, so NaN is not expected.

---

## Coordinate model (unchanged)

Backend stores top-left `x`, `y`, `width`, `height`, `rotation`.
Canvas renders each object's Group at the center: `groupX = x + width/2`, `groupY = y + height/2`.
`getTopLeftFromCenterPosition`, `calculateTransformResult`, `buildMovePatch`, `buildTransformPatch`, `formatCoordinate` are all preserved.

---

## Role / read-only behaviour

- **Members** can see the grid and toggle its visibility.
- **Members** do not see the Snap toggle or grid-size buttons.
- **Members** cannot drag, resize, rotate, or keyboard-move objects (unchanged from PR 034–035).

## Manual test checklist

1. Open floor layout page → grid visible at default 20 px spacing.
2. Toggle Show Grid off → grid disappears, objects remain.
3. Toggle Show Grid on → grid reappears.
4. Change grid size to 10 → grid spacing tightens; to 40 → spacing widens.
5. Enable Snap → drag object to a non-aligned position; release → saved position is on the nearest grid point. Inspector shows snapped values.
6. Disable Snap → drag object → exact drag-end position saved.
7. Drag object past left/top → object clamps to x=0, y=0.
8. Drag object past right/bottom → object clamps to (canvasWidth−width, canvasHeight−height).
9. Resize object very small → minimum 10 px enforced.
10. Resize object large → object doesn't expand beyond canvas bounds.
11. Rotate object → rotation still saves correctly; boundary uses unrotated box.
12. Move with arrow keys near left/top edge → stays at 0 (hold Shift: 10 px step, still clamped).
13. Shift+Arrow near right/bottom edge → clamps to max valid position.
14. Hold an arrow key → object moves once and stops (e.repeat guard). No PATCH flood.
15. Member opens layout → read-only banner; no snap/size controls; arrow keys do nothing.
16. Failed PATCH after clamped drag → error shown, original position restored.
17. Refresh page → saved clamped/snapped position persists from backend.

- **Persisted editor settings** — grid/snap state resets on page reload. localStorage persistence deferred.
